# 🛠️ Руководство по локальной разработке

## 🎯 Безопасность: Локальная разработка не влияет на продакшн

**Важно:** Ваши локальные изменения **НЕ МОГУТ** навредить продакшну на Railway, потому что:

1. ✅ **Локальная база данных** - вы используете свою локальную PostgreSQL
2. ✅ **Переменные окружения разделены** - Railway использует свои переменные из панели
3. ✅ **`.env` файл не коммитится** - он в `.gitignore`
4. ✅ **Миграции применяются только локально** - Railway применяет свои при деплое

---

## 📋 Текущая настройка

### Продакшн (Railway):
- **База данных:** PostgreSQL (Railway предоставляет `DATABASE_URL` автоматически)
- **Переменные окружения:** Настроены в панели Railway
- **Деплой:** При пуше в `main` ветку

### Локальная разработка:
- **База данных:** Локальная PostgreSQL (`localhost:5432`)
- **Переменные окружения:** Файл `.env` (не коммитится)
- **Рабочая директория:** `development` ветка

---

## ⚙️ Настройка локальной разработки

### 1. Локальная база данных PostgreSQL

Убедитесь, что у вас установлен PostgreSQL локально:

```bash
# Проверка установки
psql --version
```

### 2. Создайте локальную базу данных

```sql
-- Подключитесь к PostgreSQL
psql -U postgres

-- Создайте базу данных для локальной разработки
CREATE DATABASE employees_dev;

-- Выйдите
\q
```

### 3. Настройте `.env` файл

Файл `.env` должен содержать (он уже существует, проверьте):

```env
# Локальная разработка
NODE_ENV=development
DATABASE_URL="postgresql://postgres:ваш_пароль@localhost:5432/employees_dev"
JWT_SECRET=your-super-secret-jwt-key-here-12345
PORT=5000
```

**⚠️ ВАЖНО:** 
- Замените `ваш_пароль` на пароль вашего PostgreSQL
- Замените `employees_dev` на имя вашей локальной базы
- Этот файл **НЕ коммитится** в Git (в `.gitignore`)

### 4. Примените миграции локально

```bash
# Перегенерируйте Prisma Client
npx prisma generate

# Примените миграции к локальной базе
npx prisma migrate deploy

# Или для разработки (создаст новую миграцию при изменениях)
npx prisma migrate dev
```

## 🚀 Запуск локальной разработки

```bash
# Запуск в режиме разработки (автоперезагрузка)
npm run dev:local

# Или отдельно:
npm run server  # Сервер на http://localhost:5000
npm run client  # Клиент на http://localhost:3000
```

## 🔒 Гарантии безопасности

### Что НЕ попадет на продакшн:

1. ❌ **Локальные миграции** - Railway применяет только те миграции, которые в репозитории
2. ❌ **Локальные изменения БД** - Railway использует свою базу данных
3. ❌ **Локальный `.env`** - Railway использует переменные из панели
4. ❌ **Локальный Prisma Client** - перегенерируется при деплое

### Что попадет на продакшн:

1. ✅ **Изменения кода** (только при пуше в `main`)
2. ✅ **Миграции из `prisma/migrations`** (применяются при деплое)
3. ✅ **Изменения схемы Prisma** (влияют на продакшн)

## 📝 Рабочий процесс

### Для локальной разработки:

1. **Работайте на ветке `development`:**
   ```bash
   git checkout development
   ```

2. **Создавайте миграции для локальной БД:**
   ```bash
   npx prisma migrate dev --name название_миграции
   ```

3. **Тестируйте локально:**
   ```bash
   npm run dev:local
   ```

4. **Когда готово - коммитьте в `development`:**
   ```bash
   git add .
   git commit -m "feat: описание изменений"
   git push origin development
   ```

### Для деплоя на продакшн:

1. **Смерджите `development` в `main`:**
   ```bash
   git checkout main
   git merge development
   git push origin main
   ```

2. **Railway автоматически:**
   - Подхватит изменения
   - Пересоберет приложение
   - Применит миграции к продакшн БД
   - Перезапустит сервис

---

## ⚠️ Важные предупреждения

### НЕ делайте:

1. ❌ **Не коммитьте `.env`** - он в `.gitignore` по умолчанию
2. ❌ **Не используйте продакшн DATABASE_URL локально** - это опасно
3. ❌ **Не пушите в `main` без проверки** - сначала тестируйте в `development`

### Делайте:

1. ✅ **Тестируйте миграции локально** перед пушем
2. ✅ **Проверяйте изменения** в `development` ветке
3. ✅ **Используйте отдельную БД** для разработки

---

## 🧪 Тестирование миграций локально

Перед применением миграций на продакшн, протестируйте их локально:

```bash
# 1. Создайте миграцию
npx prisma migrate dev --name test_migration

# 2. Проверьте SQL файл миграции
# Файл: prisma/migrations/.../migration.sql

# 3. Примените к локальной БД
npx prisma migrate deploy

# 4. Проверьте что все работает
npm run dev:local

# 5. Если все ОК - коммитьте миграцию
git add prisma/migrations
git commit -m "feat: add migration"
```

## 🔍 Проверка текущей настройки

### Проверьте локальный `.env`:
```bash
cat .env
# или в PowerShell:
Get-Content .env
```

### Проверьте подключение к локальной БД:
```bash
npx prisma db pull  # Проверка подключения
```

### Проверьте Railway переменные:
- Откройте панель Railway
- Перейдите в Settings → Variables
- Проверьте `DATABASE_URL` (должен быть PostgreSQL от Railway)

   - Локально: `.env`
   - Продакшн: Railway Variables

4. **Регулярно синхронизируйте:**
   - Периодически делайте `git pull` на локальной машине
   - Синхронизируйте `development` с `main`

## ❓ Часто задаваемые вопросы

**Q: Могу ли я случайно удалить данные на продакшне?**  
A: Нет, если вы работаете локально. Railway использует свою БД, которая недоступна извне.

**Q: Как безопасно протестировать миграцию?**  
A: Сначала примените миграцию локально, проверьте, затем коммитьте и деплойте.

**Q: Что если я изменю схему Prisma?**  
A: Создайте миграцию локально, протестируйте, затем коммитьте. Railway применит миграцию при деплое.

**Q: Как откатить миграцию на продакшне?**  
A: Создайте новую миграцию для отката и задеплойте ее.
